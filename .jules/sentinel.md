## 2024-05-14 - Prevent Information Disclosure in Temporary Files
**Vulnerability:** Temporary files created using `writeFileSync` with just an encoding string (e.g., `"utf8"`) or default permissions in shared directories like `os.tmpdir()` can be read or modified by other users on the system, leading to local information disclosure or tampering.
**Learning:** Node.js file creation functions use the process's `umask` by default, which is often overly permissive (like `0o644` or `0o666`) for temporary files holding potentially sensitive data (like source code edits or diffs). We must explicitly set restrictive permissions.
**Prevention:** To prevent information disclosure vulnerabilities, always explicitly set secure file permissions (e.g., `{ mode: 0o600 }`) when creating temporary files in shared directories like `os.tmpdir()` using functions like `writeFileSync`.

## 2024-11-09 - Prevent Information Disclosure in Extension Configuration and Lock Files
**Vulnerability:** Configuration files (e.g., `nvidia-nim.json`, `settings.json`) and application state lockfiles (`.lock`) were being written using `fs.writeFile` or `fs.open` without explicitly setting file permissions. In shared environments, default `umask` configurations could make these files readable or modifiable by unauthorized users, exposing sensitive provider settings or enabling denial-of-service through lockfile manipulation.
**Learning:** Even if a secret like an API key is moved to a centralized authentication manager, configuration data and application state files still represent potentially sensitive information that should be protected by the principle of least privilege.
**Prevention:** Always pass explicit secure file permissions (e.g., `{ mode: 0o600 }` to `fs.writeFile` or `0o600` to `fs.open`) when writing configuration files, state files, or any non-public data, ensuring that access is restricted to the owner.
